% --- Executive summary ---
\section*{Summary}
Agentic language-model systems achieve strong results on reasoning and tool-use
tasks, but they typically lean on a single large model that is expensive and slow.
We built and characterized a \emph{hierarchical agentic inference system} that pairs
a small, locally-hosted model acting as a self-reflective router with a tier of
progressively larger external solver models. A server-side agent loop validates each
answer, repairs failing code, and escalates to the strongest model only when the
cheaper path fails. On the full 248-problem benchmark the system solved 96.4\% of
problems, but the striking result is that 99.6\% of all inference cost came from the
small number of calls that reached the 550B tier, while the cheapest tier solved every
one of the 220 problems it received for under half a percent of the cost. A live
on-TPU run validates the same pipeline end-to-end, and re-pricing shows that defaulting
to the strongest model would cost roughly $2{,}600\times$ more for no accuracy gain. We
also characterized the local router under vLLM and studied solver-side prompt
optimizations, finding that token-cutting tricks can be net cost-negative once their
prompt overhead and escalation side effects are accounted for. The dominant cost lever
is therefore routing accuracy---keeping problems off the expensive tier---rather than
router speed or token thrift.

% --- Section 1: Introduction ---
\section{Introduction}
\label{sec:introduction}
Traditional LLM interactions are single- or multi-turn chats against one model.
Agentic applications, in contrast, decompose a goal into steps, invoke tools,
validate intermediate results, and reflect on failures. The dominant implementation
choice today is to attach this control flow to a single high-end model: the agent is
smart because the underlying model is, and the system simply absorbs the cost. That
choice has two consequences. At the \emph{application} level, every problem---trivial
or hard---pays the same per-token rate, which is wasteful when most problems are easy.
At the \emph{systems} level, the inference engine spends most of its decode budget
generating long reasoning traces on a model that is over-provisioned for the task,
while small, cheap models sit idle.

This project takes the opposite stance. We treat \emph{which model runs} as the
primary control variable of the agent and ask how cheaply we can solve a mixed
benchmark if the agent itself is responsible for picking the right solver. The design
pattern is \emph{dynamic model orchestration}: a small, locally-hosted model serves
as a router that reads the problem, picks the cheapest external solver likely to
succeed, and emits a structured decision; a server-owned agent loop then drives that
solver through validation, tool-assisted repair, and, if necessary, a single
escalation to the strongest available model. The router is itself an agent step---it
self-critiques in a second pass and may revise its choice before any expensive solver
call is made.

To quantify effectiveness, we introduce an analytical cost function calibrated to
inference-engine behavior and apply it to every model call the agent makes. The
function attributes cost to the specific decisions the agent took---which model was
selected, how many prompt and completion tokens it consumed, how many tool
invocations and attempts the problem required, and whether escalation fired---so a
run's aggregate cost can be decomposed and compared against any other run on the same
benchmark.

% --- Section 2: System Architecture ---
\section{System Design and Implementation}
\label{sec:architecture}
This section describes the problem domain we target, the agentic pattern we
implement, the tool and state machinery the agent relies on, and the two-tier
software architecture that hosts it.

\subsection{Problem Domain and Use Case}
\label{sec:domain}
The target application is a general-purpose autonomous reasoning agent that answers a
heterogeneous stream of benchmark problems under a hard constraint on aggregate
inference cost. Each problem carries enough metadata for the system to route, solve,
and grade it without human intervention (Table~\ref{tab:problem-schema}).

\begin{table}[ht]
\centering
\small
\begin{tabular}{@{}lp{0.74\linewidth}@{}}
\toprule
\textbf{Field} & \textbf{Description} \\
\midrule
\texttt{problem\_id} & Unique integer identifier. \\
\texttt{problem}     & The problem text shown to the solver. \\
\texttt{difficulty}  & One of \texttt{very\_easy}, \texttt{easy}, \texttt{medium}, \texttt{hard}. \\
\texttt{category}    & One of \texttt{math}, \texttt{code}, \texttt{factual}, \texttt{reasoning}, \texttt{web}, \texttt{image}. \\
\texttt{verify}      & Validation mode: \texttt{match} (exact string), \texttt{tests} (asserts run against the model's code), \texttt{judge} (LLM-graded), or \texttt{heuristic} (named server-side validator). \\
\bottomrule
\end{tabular}
\caption{Per-problem metadata schema.}
\label{tab:problem-schema}
\end{table}

The benchmark assembled for this project contains 248 problems: 164 code problems
from HumanEval, 40 math problems from GSM8K, and a curated set of $\sim\!44$ math,
reasoning, factual, and image problems. Problems are loaded per domain, so a run can
target one domain or their union. The workload is deliberately non-trivial: code
problems must execute against unit tests in a sandboxed interpreter, web problems
require fetching and conditioning external context before answering, and arithmetic or
factual problems range from trivial lookups to multi-step reasoning. A single model
choice cannot serve this mix efficiently, so the agent's job is to land each problem
on the cheapest tier that will solve it.

\subsection{Agentic Pattern: Dynamic Model Orchestration}
\label{sec:pattern}
We implement dynamic model orchestration as the primary pattern, with reflection
layered into both the router and the solver. The agent is realized as two cooperating
loops.

\textbf{Router loop.}
The locally-hosted Granite-4.1-3B model receives a system prompt listing the four
external solver tiers, each annotated with its parameter counts and a plain-English
description of the reasoner it represents (high schooler, second-year university
student, entry-level worker, PhD student). The router emits a structured decision
consisting of a brief reasoning trace and a chosen \texttt{model\_id}. Because
Granite-4.1-3B has no inherent chain-of-thought behavior, requiring it to produce the
reasoning field \emph{before} the choice forces a prompt-induced chain-of-thought. A
second turn then asks the router to inspect its own reasoning and decide whether the
initial rationale was correct, allowing it to revise its decision. The server executes
the revised \texttt{model\_id} and records the updated reasoning. If the decision is
unparseable, the router fails safe to the strongest model rather than dropping the
problem.

\textbf{Solver loop.}
The server's \texttt{solve\_problem} routine drives the chosen external model through
up to \texttt{max\_attempts} tries. For each try it (1) prepares tool context (e.g.,
fetches and truncates the web page for a \texttt{web} problem), (2) builds the solver
chat with format-specific answer instructions, (3) calls the model, and (4) validates
the answer in the mode the problem requires---executing candidate code against its
assertions in the sandboxed runner for \texttt{verify=tests} problems. On failure, the
server appends the runner's stdout/stderr back to the model so it can repair its code
on the next attempt. If all attempts fail, the server escalates exactly once to the
strongest registered model and reruns validation. This one-shot escalation cap keeps
the worst-case cost bounded.

\textbf{State and tools.}
For each problem the agent maintains the running conversation, the tools invoked, the
tokens and cost accumulated, and the size of any injected external context. State is
held in memory for the duration of the problem and persisted at the end; no agent-side
state survives across problems, which keeps the service horizontally scalable. The
agent has two deterministic, server-driven tools rather than model-native function
calls: \texttt{web\_search} fetches a problem's \texttt{source\_url}, optionally
compresses the body, and caps it at $10{,}000$ characters before it enters the prompt;
\texttt{run\_python\_code} is a sandboxed executor whose output feeds the code-repair
turn.

\subsection{Two-Tier Software Architecture}
\label{sec:sysarch}
\textbf{Local inference tier.}
The local router is IBM Granite-4.1-3B served behind a vLLM OpenAI-compatible endpoint
and exercised concurrently by an \texttt{asyncio} driver that keeps multiple router
requests in flight. This is what lets the engine see a decode-heavy, batchable
workload rather than serialized one-at-a-time calls---a prerequisite for the
systems-level optimizations we evaluate in Section~\ref{sec:eval}.

\textbf{External solver tier.}
Four models are exposed through an OpenAI-compatible provider (OpenRouter in our
setup), ordered into progressively stronger tiers primarily by active parameter count
and expected reasoning ability (Table~\ref{tab:solvers}). This gives the router a
discrete set of cost--capability tradeoffs. Active parameter count is the right axis
because three of the four solvers are mixture-of-experts models, whose serving cost
tracks active rather than total parameters. Post-training also matters when
interpreting the tiers: the strongest tier is treated as stronger not only because of
its larger scale but because it is expected to have undergone a more extensive
reasoning-oriented post-training regime, which is why the gap between the top two
tiers is not explained by parameter count alone.

\begin{table}[ht]
\centering
\small
\begin{tabular}{@{}llr@{}}
\toprule
\textbf{Tier} & \textbf{Model} & \textbf{Total / Active params} \\
\midrule
Small (cheapest) & OpenAI GPT-OSS-20B   & 21B / 3.6B \\
Medium           & OpenAI GPT-OSS-120B  & 117B / 5.1B \\
Large            & Qwen3-Coder          & 480B / 35B \\
Strongest        & NVIDIA Nemotron-3-Ultra & 550B / 55B \\
\bottomrule
\end{tabular}
\caption{External solver tiers, ordered by capability. The strongest tier is the

\label{tab:solvers}
\end{table}

\textbf{Agent service.}
A FastAPI service mediates between the two tiers and owns everything that is not
inference: tool execution, validation, the cost function, and metrics persistence.
Each run is registered ahead of time with a unique \texttt{run\_id} and a fixed set of
optimization flags, which the service treats as the authoritative configuration for
everything it logs.

\subsection{Optimizations}
\label{sec:optimizations}
Each run is registered with a fixed set of boolean optimization flags, grouped by
where they take effect:
\begin{itemize}
  \item \textbf{Inference-engine:} \texttt{quantized\_local\_lm}
        (BF16\,$\leftrightarrow$\,FP8 weights) and \texttt{quantized\_kv\_cache}
        (FP8 KV cache), both targeting the local router's throughput and memory.
  \item \textbf{Prompt and context:} \texttt{caveman} (a terse-rewrite instruction
        meant to cut completion tokens), \texttt{web\_search\_compression} (caps web
        context size), and two long-context compressors---a deterministic
        head-and-tail variant and an AI-driven variant that falls back to the
        deterministic one on any error.
  \item \textbf{Routing policy:} \texttt{capabilities\_prompt} adds per-model
        benchmark scores to the router prompt, and \texttt{local\_model\_solves} lets
        the router short-circuit trivial \texttt{very\_easy} / \texttt{match}
        problems without any external call.
\end{itemize}
Flags are independently toggleable against a \texttt{baseline} run with all of them
disabled, so the cost function can attribute any movement to a single optimization or
a stack of them.

% --- Section 3: Characterization Framework ---
\section{Performance Characterization Framework}
\label{sec:characterization}
We characterize the system at two levels. The \emph{system level} measures the
efficiency of the local routing infrastructure---throughput, latency, and memory of
the vLLM-served router. The \emph{application level} measures the effectiveness of the
complete agentic workflow---solve rate, routing quality, and the cost--performance
tradeoff. The two are linked: a faster but less accurate router can lower local
serving cost while \emph{raising} total application cost through extra retries and
escalations, so an optimization is only deemed beneficial if it improves the overall
cost-vs-quality frontier.

\subsection{Cost Model}
\label{sec:costmodel}
Because provider API prices are VC-subsidized and would simply reward routing
everything to the largest model, we score every model call with an analytical cost
function tied to inference-engine behavior. For a call with $p$ prompt tokens and $c$
completion tokens on a model whose active-parameter scale (relative to the smallest
solver) is $s$, the per-call cost combines a quadratic attention prefill, a linear MLP
term, and a decode term, modulated by a mild hardware penalty on total size $T$:
\begin{equation}
\label{eq:cost}
C = \underbrace{\big(\alpha s^2 + (1-\alpha)s\big)\,T^{1.2}}_{\text{model factor}}
    \cdot \big[\, k_{\text{in}}\,f(p) + k_{\text{out}}\,c\,\bar{\ell} \,\big],
\end{equation}
where $\alpha$ is the attention fraction of weights, $f(p)$ is the quadratic prefill
term, and $\bar{\ell}$ is the average sequence length during decode. A run's total
cost is the sum of Eq.~\eqref{eq:cost} over every router attempt, solver attempt,
repair, and escalation, accumulated per problem. This makes cost an explicit function
of the agent's decisions rather than a flat per-token rate.

\subsection{System- and Application-Level Metrics}
\label{sec:metrics}
The router sits on the critical path of every problem, so at the \emph{system level} we
report decode throughput (output tokens/s) and serving capacity (requests/s), plus
time-to-first-token (TTFT), peak HBM memory (which bounds batch size), and batch
efficiency (batched vs.\ single-request throughput, indicating how well vLLM batches
the routing workload). At the \emph{application level} the primary metric is
\emph{solve rate}, paired with \emph{aggregate cost} from Eq.~\eqref{eq:cost}, plus
\emph{escalation rate}, \emph{average attempts} per problem, and router calibration. For
calibration we define a target tier per difficulty
(\texttt{easy}\,$\rightarrow$\,smallest, \texttt{medium}\,$\rightarrow$\,mid,
\texttt{hard}\,$\rightarrow$\,top) and report \emph{router accuracy} (correct initial
routes / total) with its two error modes: \emph{over-routing} (routed too high) and
\emph{under-routing} (routed too low). Over-routing preserves solve rate but wastes cost
on unnecessarily
capable models; under-routing is costlier still because it triggers retries and,
ultimately, escalation.

% --- Section 4: Performance Evaluation ---
\section{Performance Evaluation}
\label{sec:eval}

\subsection{Experimental Setup}
\label{sec:setup}
We draw on four experiments plus a throughput profile, all using the analytical cost
function of Eq.~\eqref{eq:cost} in custom units (not provider prices). The
\textbf{full benchmark} (\texttt{baseline\_001}) runs all 247 logged problems with real
OpenRouter solver calls, difficulty-based routing, and escalation enabled---the only
run that exercises the expensive tiers, and thus where we read the cost story. The
\textbf{live on-TPU} runs (\texttt{tpu\_bench}) use the real Granite-4.1-3B router
served by vLLM (BF16) on a Google Cloud TPU v5e (16\,GB HBM), driving 134 problems
end-to-end. The \textbf{forced single-model eval} pins each lineup model as a standalone
solver (no escalation) over a balanced 32-problem slice to measure real per-model
accuracy. The \textbf{solver-side study} runs four prompt configurations over the same
66 problems with the router held fixed. Validation uses numeric matching for math,
curated validators for reasoning/factual, and \texttt{verify=tests} (candidate run
against its asserts, with a solve/run/repair loop) for code.

\subsection{Full Benchmark: Where the Cost Goes}
\label{sec:fullbench}
On the full 247-problem benchmark the agent solved \textbf{238 (96.4\%)} at an
aggregate cost of \textbf{0.706} custom units, averaging 1.11 attempts per problem.
The interesting result is not the solve rate but \emph{where the cost lives}
(Table~\ref{tab:routing-outcome}). The router sent 220 problems to the cheapest tier
and that tier solved \textbf{all 220}, for 0.4\% of the run's cost. Almost the entire
0.706---\textbf{99.6\%}---came from the \textbf{25 calls that reached the 550B tier},
which solved only 16 of them. In other words, the cheap tier carries the workload
essentially for free, and the strong tier is simultaneously the cost sink and the
place where problems still fail.

\centering
\small
\begin{tabular}{@{}lrrrr@{}}

\midrule
GPT-OSS-20B (cheap)     & 220 & 220 & 0.0030 & 0.4\% \\
GPT-OSS-120B (medium)   & 2   & 2   & 0.0002 & 0.0\% \\
Nemotron-550B (strongest) & 25 & 16  & 0.7031 & 99.6\% \\
\midrule

\label{tab:routing-outcome}

Router calibration on this run is good but not perfect: 4.0\% of problems were
under-routed (escalated because the first choice was too weak) and 2.0\% over-routed
(an easy problem solved by an unnecessarily expensive tier), leaving $\sim\!94\%$
appropriately routed and 10 escalations total. The cost of those errors is visible by
difficulty: \texttt{easy} problems cost 0.216 in aggregate---\emph{more} than the 0.133
spent on \texttt{hard} problems---because a handful of easy code problems were
mis-routed or escalated to the 550B tier, each costing 0.03--0.09. This is the concrete
form of the thesis: a single mis-routed easy problem on the expensive tier costs more
than dozens of correctly-routed hard ones.

\subsection{How Much Does the Model Choice Matter?}
\label{sec:modelchoice}
Two views confirm that the cheap tier is sufficient and that defaulting to a larger
model buys nothing but cost. First, re-pricing the live 134-problem workload (same
observed tokens) on each solver (Figure~\ref{fig:router_vs_models}): the cheapest tier
is the cost floor, the medium tier costs $\sim\!12\times$, and the 550B model costs
$\sim\!2{,}600\times$, with no solve-rate headroom to gain. Second, running each model
as a \emph{standalone} solver on a balanced 32-problem slice (Table~\ref{tab:forced})
measures this empirically rather than by projection.

The empirical spread is tiny on this mix: GPT-OSS-120B (100\%) edges GPT-OSS-20B
(96.9\%) by $\sim$3 points for $\sim\!12\times$ the cost per problem, and the strongest
model, Nemotron-550B, actually solves \emph{fewer} (95.0\%) than the cheap 20B while
costing $\sim\!1{,}790\times$ as much per problem and running $>$120\,s/call.
The router's own cost/problem sits essentially at the cheap-tier floor
($\sim\!1.4\times$). Its one weak spot is robustness rather than routing: the on-device
two-pass granite decision dropped 5/32 problems (non-JSON or timeout) \emph{before} they
reached a solver, so end-to-end it trailed simply always using the cheap tier---a
router-stage reliability cost, fixable by hardening the JSON parse and gating the
self-critique pass on low confidence.

\subsection{Live On-TPU Validation}
\label{sec:live}
The \texttt{tpu\_bench} runs confirm the full pipeline works on real hardware: the
Granite router on the TPU v5e drove 134 problems across four domains end-to-end through
OpenRouter solvers, solving \textbf{121}. Because this sample skews easy/medium, the
router sent every reached problem to the cheapest tier with \textbf{zero escalations},
and that tier solved every problem it received; the 13 non-solves were all free-tier
rate-limits (HTTP 429) or one non-JSON router reply---infrastructure, not capability.
This is the existence proof that the on-device router, resolution server, validation,
cost accounting, and metrics work together live; the cost spread it cannot exercise is
exactly what the full-benchmark run (Section~\ref{sec:fullbench}) provides.

\subsection{System Level: Router Serving Throughput}
\label{sec:throughput}
Cost characterizes what we pay per solved problem; throughput characterizes how fast
the on-device router serves. We profiled the live BF16 router with a synthetic
fixed-length workload (48 requests, concurrency 6, 512-token prompts, 256 output tokens
with \texttt{ignore\_eos} so every request emits exactly \texttt{max\_tokens}). It
sustains \textbf{$\sim\!590$ output tok/s} at concurrency 6 ($\sim\!101$ tok/s per
request), \textbf{2.30 requests/s}, a 2.57\,s p50 latency, and a \textbf{$\sim\!46$\,ms
TTFT}. Because the router only emits a short JSON decision, its throughput matters
mostly as latency overhead before a solver is dispatched ($\sim\!46$\,ms TTFT plus
$\sim\!2.5$\,s for the two-pass decision). On TPU v5e, \texttt{DTYPE=fp8} maps to INT8
weight quantization (v5e has no FP8 weights) while FP8 \emph{KV cache} is supported; the
quantization sweep restarts the router per config and is pending a maintenance window on
the shared host.

\subsection{Solver-Side Optimization Study}
\label{sec:optstudy}
The live runs measure \emph{routing}. To isolate \emph{solver-side prompt}
optimizations on a clean, comparable matrix, we ran four configurations over the same
66 problems, holding the router fixed as a deterministic difficulty$\rightarrow$model
stand-in so the only variable is the prompt treatment. To keep the optimization signal
from being drowned out, the top tier's escalation cost was re-tuned so a single
escalation costs $\sim\!2.7\times$ the large tier rather than $\sim\!99\%$ of a run.
Table~\ref{tab:optstudy} and Figure~\ref{fig:opt_decomp} summarize the outcome.

The headline is that the seemingly obvious token-cutting optimization is the worst one.
\texttt{caveman} rewrites the solver prompt to demand terse answers and does shrink

\texttt{combo} that includes it inherits the loss, while \texttt{longctx\_ai} is the
only configuration that comes in below baseline, and only marginally.

problems are 58\% of \texttt{caveman}'s total cost versus 35\% for the baseline.

the format check, and ends up costing $\sim\!\$0.038$ and unsolved---a roughly
$3{,}700\times$ blow-up on a problem that should be free.

% --- Section 5: Optimization Opportunities ---
\section{Optimization Opportunities}
\label{sec:optimization}
The evaluation points to a clear ordering of where effort pays off.

\textbf{Routing accuracy dominates everything.}
The solver tier drives both cost and latency: on the full benchmark 99.6\% of all cost
came from the 25 calls that reached the 550B tier (Table~\ref{tab:routing-outcome}),
and re-pricing shows a $\sim\!2{,}600\times$ spread between the cheapest and strongest
solver (Figure~\ref{fig:router_vs_models}), whereas router serving overhead is
milliseconds. The single largest deployment win is therefore avoiding unnecessary
escalation and over-routing, not making the router faster. The \texttt{caveman} study
reinforces this from the opposite direction: an optimization that is locally sensible
(fewer completion tokens) is globally harmful because it perturbs routing and escalation
behavior.

\textbf{Optimize for the workload, not the token.}
\texttt{caveman}'s fixed per-call prompt overhead looks worst exactly where problems
are smallest, and \texttt{longctx\_ai} never crosses its compression threshold on
short prompts, making it an honest no-op here. Both signs could flip on a long-context
workload---the lesson is that prompt optimizations must be matched to the problem
distribution, and gated (e.g., by length or difficulty) rather than applied
unconditionally.

\textbf{Router-tier systems wins are second-order.}
On the local tier, weight + KV-cache quantization (the pending sweep) frees HBM for a
larger \texttt{max\_num\_seqs}, prefix caching already amortizes the identical routing
prompt across problems, and the two-pass self-revision could be gated on low confidence
to halve router latency. These help latency and serving density but, per the first
point, are second-order to routing accuracy for total cost. Once routing is accurate and
the router is quantized and batched, the remaining cost is intrinsic solver compute on
genuinely hard problems---the next frontier being techniques like VLM-describe routing
that convert expensive image tokens to a cheap text description before the large model
sees them.

% --- Section 6: Conclusion ---
\section{Conclusion}
\label{sec:conclusion}
We built and characterized a hierarchical agentic inference system in which a small,
locally-hosted model routes each problem to the cheapest external solver likely to
succeed, backed by a server-owned loop that validates, repairs, and escalates within a
bounded budget. On the full 248-problem benchmark it solved 96.4\% of problems, and the
cost decomposition is unambiguous: the cheapest tier solved all 220 problems it
received for 0.4\% of the cost, while 99.6\% of the cost came from the few calls that
reached the 550B tier---which also accounted for most of the remaining failures. A live
on-TPU run validated the same pipeline end-to-end, a forced single-model study showed
the capability spread between tiers is only a few points on this mix, and a controlled
study showed that token-cutting prompt optimizations can be net cost-negative once
prompt overhead and escalation side effects are counted. Together these reinforce that
routing accuracy---keeping problems off the expensive tier---rather than token thrift or
router speed is the dominant cost lever. The main limitation is that our benchmark skews
easy/medium, so the expensive tiers are exercised mainly by mis-routes; the clear next
step is a harder problem set that forces capability-driven escalation, together with the
pending router quantization sweep and VLM-describe routing for the image tier.
